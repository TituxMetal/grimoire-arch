### 2. GitHub Pages deploy — `site` / `base` (the footgun)
Source: [Deploy Astro to GitHub Pages](https://docs.astro.build/en/guides/deploy/github/).
- **`site`**: set to the Pages URL, `https://<username>.github.io`.
- **`base`**: **required when the repo is NOT named `<username>.github.io`** — set it to `/<github-repo-name>` (leading slash). **CRITICAL: `base` must match the GitHub repo name, NOT the local folder name.** Local convention prefixes the folder with a creation date (`20260602-grimoire-arch`), but the GitHub repo will be named **`grimoire-arch`** (no date prefix). So the site lives at `https://<user>.github.io/grimoire-arch/` and **`base: '/grimoire-arch'` is required** — do NOT use the dated local folder name. Omit `base` only if you create a dedicated `<user>.github.io` repo or attach a custom domain. Confirm the exact repo name at `git remote add` time and set `base` to match it character-for-character.
- **Base-path footgun (directly threatens the proof-slice):** internal links and asset URLs must respect `base`. Starlight resolves its own nav with `base`, but hand-authored relative links like the `guide/annexe-b → ../adr/` seam must resolve correctly under the subpath. `bun run build` is the canary — it errors on broken internal links. Verify the deployed subpath, not just localhost root. Note the subpath is `/grimoire-arch/` (the GitHub repo name), not the dated local folder name.

### 3. The deploy workflow — official action, bun-aware
Source: [withastro/action](https://github.com/withastro/action) (current **v6.1.1**, Apr 2026; pin `@v6`).
- **The official action natively supports bun**: it auto-detects the package manager from the lockfile (`bun.lock`/`bun.lockb`) and runs install + `<pm> run build`. Supported PMs: npm, yarn, pnpm, **bun**, deno. You can force it with `package-manager: bun@latest`.
- **Open decision for an ADR (`docs/adr/`):** two viable paths, **both fully "bun only"** (neither uses npm/pnpm/yarn, so both satisfy the real doctrine constraint):
  - **(A) `withastro/action@v6`** — least config; it auto-detects the bun lockfile and runs `bun install` + `bun run build` automatically. Simplest, fewest moving parts. **Recommended default for a docs site.**
  - **(B) Manual workflow with `oven-sh/setup-bun@v2`** — explicit `bun install` + `bun run build` + `actions/upload-pages-artifact` + `actions/deploy-pages`. More verbose, more control, more surface to maintain. Source: [oven-sh/setup-bun](https://github.com/oven-sh/setup-bun).
  - **Re the AGENTS.md mention of `setup-bun`:** that line reads as drafting-time phrasing, not a locked brainstorm decision — it was never an explicit user choice. The binding constraint is "bun only," which (A) honors just as well. So the choice is **genuinely open**; default to (A) for simplicity unless there's a concrete reason to hand-roll the pipeline. Record whichever you pick in an ADR. Either way commit the bun lockfile.
- Verbatim official `withastro/action` workflow (adapt `path`/`build-cmd` per choice):
  ```yaml
  name: Deploy to GitHub Pages
  on:
    push:
      branches: [main]
    workflow_dispatch:
  permissions:
    contents: read
    pages: write
    id-token: write
  jobs:
    build:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v6
        - uses: withastro/action@v6
          # with:
          #   package-manager: bun@latest
          #   build-cmd: bun run build
    deploy:
      needs: build
      runs-on: ubuntu-latest
      environment:
        name: github-pages
        url: ${{ steps.deployment.outputs.page_url }}
      steps:
        - id: deployment
          uses: actions/deploy-pages@v4
  ```
- **Required perms** (must be set or deploy 403s): `contents: read`, `pages: write`, `id-token: write`.
- **One-time repo setting:** GitHub repo → Settings → Pages → Source = **GitHub Actions** (not "Deploy from branch"). Easy to forget; blocks Done-criterion #3.

## TypeScript version (TS 6 vs TS 7 beta)
Facts dated **as of 2026-06-02**.

### 1. Current stable TS = 6.0; TS 7 is **beta-only**, not stable
- **TS 6.0 is the current stable line** and the **last JavaScript-based** release — a deliberate stepping-stone before the Go rewrite. Source: [Announcing TypeScript 6.0](https://devblogs.microsoft.com/typescript/announcing-typescript-6-0/).
- **TS 7 = the native Go port ("tsgo", Project Corsa), ~10x faster.** Reached **beta on 2026-04-21 — beta, not stable.** Ships as a **separate package `@typescript/native-preview`** with the **`tsgo`** entry point (not the `typescript`/`tsc` package). Stable TS 7 (which will take over the `typescript` package + `tsc`) was signposted ~mid-2026 but **has no confirmed GA date** as of 2026-06-02. Sources: [Announcing TypeScript 7.0 Beta](https://devblogs.microsoft.com/typescript/announcing-typescript-7-0-beta/), [typescript-go repo](https://github.com/microsoft/typescript-go). **Confidence: high** on "beta"; **medium** on GA timing — don't quote a release date.
- `tsgo` targets behavioral parity with `tsc` from **TS 6.0** — so TS 6 remains the semantic baseline.

### 2. What Astro 6 expects
- `bun create astro` pins **`typescript`** in `devDependencies` and writes a `tsconfig.json` extending an Astro preset (`astro/tsconfigs/base|strict|strictest`). Astro 6's tsconfig template was updated to prep for **TS 6** (adds `"node"` to `compilerOptions.types`). Source: [Astro TypeScript guide](https://docs.astro.build/en/guides/typescript/).
- **Footgun (medium confidence):** `@astrojs/check@0.9.8` declares peer dep **`typescript@^5.0.0`**, which conflicts with `typescript@^6` under Astro 6 and emits peer-range warnings. Tracked: [astro#16112](https://github.com/withastro/astro/issues/16112). Let the scaffold pin versions; a peer-range warning from `astro check` is this known lag, not a real break.

### 3. Does Astro/Starlight support the TS 7 native compiler (tsgo)?
- **Not yet for `astro check`.** Astro tracks tsgo compat in [roadmap #1321](https://github.com/withastro/roadmap/discussions/1321), but `astro check` / `@astrojs/check` / `@astrojs/language-server` / `@astrojs/ts-plugin` all depend on TypeScript's programmatic Language Service API, which **tsgo does not yet expose**. No official "use tsgo" recommendation exists. **Verdict: stay on stable TS 6.x; do not adopt `@typescript/native-preview`/tsgo here.** Confidence: high.

### 4. Pragmatic verdict for grimoire-arch
A Starlight docs site is **MDX + `astro.config.mjs`**, with essentially no hand-written `.ts` — **the TS version is near-irrelevant to the build.** `bun run build` (the link canary) does not run `tsc`/`tsgo`. **Take whatever `typescript` version `bun create astro --template starlight` pins (TS 6.x line), don't add `astro check` to CI, and ignore TS 7/tsgo entirely for now.**

## Theming & styling (default theme, customization, Tailwind v4)
*All facts as of 2026-06-02.*

- **Default look — ships complete, no design-from-scratch.** Starlight provides a complete, opinionated default theme out of the box: responsive layout, built-in light/dark toggle, accessibility-conscious defaults (incl. a `contrastLevel` option for WCAG color contrast), and a default code-block syntax theme (Night Owl, dark+light). You do **not** build the look yourself. Sources: [CSS & Styling guide](https://starlight.astro.build/guides/css-and-tailwind/), [Getting Started](https://starlight.astro.build/getting-started/).
- **Customization — CSS custom properties + `customCss`.** Theming overrides Starlight's `--sl-color-*` design tokens: an **accent-color** group (links, current nav) and a **gray-shade** group (text/bg/borders). Point Starlight at your stylesheet via `customCss: ['./src/styles/custom.css']` and override tokens there. Light/dark toggle is built in. Sources: [CSS & Styling guide](https://starlight.astro.build/guides/css-and-tailwind/), [Config reference](https://starlight.astro.build/reference/configuration/).
- **Component overrides.** For what CSS can't reach, the `components` config maps a Starlight component name (`Header`, `Sidebar`, `ThemeSelect`, `LanguageSelect`, …) to your own `.astro` file. Sources: [Overrides reference](https://starlight.astro.build/reference/overrides/), [Overriding components](https://starlight.astro.build/guides/overriding-components/).
- **Community themes.** A theme = a Starlight plugin (custom CSS + overrides) you install. Browsable catalogue with live demos (themes inspired by VS Code Vitesse, Obsidian Publish, shadcn docs, Next.js docs). Source: [Themes](https://starlight.astro.build/resources/themes/), [Plugins](https://starlight.astro.build/resources/plugins/). *(Exact package names — medium confidence; the catalogue itself is confirmed.)*
- **Tailwind v4 wiring.** Native tooling is plain CSS + custom properties (no Tailwind needed). The **deprecated** path is the `@astrojs/tailwind` integration — do NOT use it for v4. Current setup uses Tailwind v4's **`@tailwindcss/vite`** plugin (`astro add tailwind`) + the **`@astrojs/starlight-tailwind`** compat CSS. CSS-first: `src/styles/global.css` does `@import "tailwindcss";`, imports Starlight's compat CSS, customizes via `@theme`, and goes **first in `customCss`**. Sources: [CSS & Styling guide](https://starlight.astro.build/guides/css-and-tailwind/), [Tailwind Astro guide](https://tailwindcss.com/docs/installation/framework-guides/astro).
- **Verdict for this project:** **skip Tailwind. Default theme + a small `customCss`** (accent color, a few token tweaks, optionally `contrastLevel`) — a ~10-line stylesheet, zero extra build surface, keeps the bun-only low-dependency footprint. Tailwind v4 only pays off for bespoke marketing/landing pages or many custom layouts *outside* the doc-content flow; for linear guide chapters + sidebar nav, Starlight's defaults already cover it.
