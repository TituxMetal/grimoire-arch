# Publier un dossier docs/ en Astro Starlight sur GitHub Pages avec bun

Recette validée bout-en-bout sur le proof-slice grimoire-arch (commits
`92d5ea3..0c85222`). Réutilisable pour tout site docs Starlight déployé sur un
**sous-chemin** GitHub Pages (`https://<user>.github.io/<repo>/`) en toolchain
**bun**. Les trois pièges ci-dessous sont **silencieux** : aucun ne fait échouer
le build tel quel, donc on les découvre en prod si on ne les connaît pas.

## La recette en bref

1. `bun create astro --template starlight` (forme bun-native).
2. `astro.config.mjs` : `site: 'https://<user>.github.io'` + `base: '/<repo>'`
   (**le nom du repo, pas un dossier local daté** — cf. ADR base-path).
3. Contenu publié sous `src/content/docs/` uniquement ; chaque `.md` a un
   frontmatter `title:` (sinon le schéma de collection Starlight casse le build).
4. Déploiement : `.github/workflows/deploy.yml` = `actions/checkout@v6` →
   `withastro/action@v6` (détecte `bun.lock`, lance `bun install` + `bun run build`)
   → `actions/deploy-pages@v4`. Permissions exactes : `contents:read, pages:write,
   id-token:write`. Triggers : `push` sur `main` + `workflow_dispatch`.
5. **Activer Pages côté repo, Source = GitHub Actions** (sinon le deploy échoue
   silencieusement, site jamais visible) :
   `gh api -X POST repos/<user>/<repo>/pages -f build_type=workflow`.
6. Vérifier **sur le vrai sous-chemin** `/<repo>/` (la correction de `base` ne se
   voit pas sur `localhost:4321/` ni à la racine du domaine).

Note credential : pour `git push` HTTPS après `gh repo create`, lancer une fois
`gh auth setup-git` (sinon `could not read Username for https://github.com`).

## Piège 1 — le canari « build casse sur lien mort » N'EXISTE PAS par défaut

### Problème
Le build `bun run build` reste **vert** même quand des liens internes sont cassés.
Pire : Astro 6 **ne réécrit pas** les liens Markdown relatifs (`../adr/x.md` fuit
tel quel dans le HTML), donc les liens sont cassés en prod sur deux axes —
profondeur (un `..` manquant sous le sous-chemin) et extension (`.md` au lieu de
l'URL-dossier).

### Cause
Avec la collection `docsLoader` de Starlight, les `.md` passent par un loader et
une route catch-all — pas de correspondance fichier→route, donc Astro ne réécrit
pas les liens relatifs. Et aucun validateur de liens n'est livré par défaut.

### Solution
Installer le vrai canari et corriger la forme des liens :
```bash
bun add -d starlight-links-validator
```
```js
// astro.config.mjs → starlight({ plugins: [starlightLinksValidator()] })
```
**Forme de lien obligatoire : root-relative AVEC base** → `/<repo>/adr/<slug>/`.
C'est la **seule** forme à la fois (a) validée par le plugin et (b) correcte en
prod. Testé empiriquement :
- `../foo.md` : fuit, cassée.
- `../../foo/` (relative corrigée) : correcte en browser mais le plugin **ignore**
  les liens relatifs (même avec `errorOnRelativeLinks:false` il les saute, ne les
  valide pas → un lien `9999` cassé passe inaperçu).
- `/foo/` (root sans base) : Starlight ne préfixe **pas** `base` aux liens Markdown
  → cassée sur le sous-chemin.
- `/<repo>/foo/` (root avec base) : ✅ validée ET correcte. Bonus : un changement
  de `base` est rattrapé par le plugin.

Décision complète : `../adr/2026-06-03-link-validation-canary.md`.

### Pour la prochaine fois
Ne jamais croire « le build docs casse sur lien mort » sans le prouver. Casser un
lien exprès et vérifier que `bun run build` rougit. Si non → pas de canari, ajouter
le plugin **avant** de propager du contenu.

## Piège 2 — Pagefind indexe en anglais un site français

### Problème
`dist/pagefind/pagefind-entry.json` montre `languages: { en: {...} }` sur un site
100 % français. Recherche accentuée de qualité dégradée (segmentation/stemming `en`).

### Cause
Starlight émet `<html lang="en">` par défaut. Pagefind choisit sa langue d'index
d'après l'attribut `lang` de la page.

### Solution
Configurer la **locale racine** monolingue (pas de préfixe d'URL) :
```js
// astro.config.mjs → starlight({...})
defaultLocale: 'root',
locales: { root: { label: 'Français', lang: 'fr' } },
```
Après rebuild : live `lang="fr"`, `pagefind-entry.json` → `languages: { fr: {...} }`.

### Pour la prochaine fois
Vérifier `<html lang>` et la clé `languages` de `pagefind-entry.json` dès le premier
build d'un site non-anglophone. Pagefind = recherche **navigateur** (WASM) : pas
d'API Node de search, et `astro dev` n'a pas d'index (construit au `build`) →
l'avertissement « search prod-only » en dev est normal.

## Piège 3 — Starlight v0.39 a supprimé les groupes sidebar autogénérés labellisés

### Problème
Build refusé : `Invalid config — Found an autogenerate object with a label. Support
for autogenerated sidebar groups was removed in Starlight v0.39.0.` puis
`Expected { link } | { items } | { slug } | string`.

### Cause
Avant v0.39 on pouvait faire un groupe `{ label, autogenerate: { directory } }`
(ou un item `{ label, autogenerate }`). v0.39 a retiré cette forme.

### Solution
Mettre l'`autogenerate` **dans `items`**, **sans `label`** sur l'objet autogenerate :
```js
{
  label: 'Coulisses / journal',
  collapsed: true,
  items: [{ autogenerate: { directory: 'adr' } }],
}
```
Le `label` du groupe reste, c'est l'objet `autogenerate` qui ne doit pas en porter.

### Pour la prochaine fois
Pour un groupe autogénéré en Starlight ≥ 0.39 : `{ label, items: [{ autogenerate:
{ directory } }] }`. Les items explicites (ordre de lecture) restent `{ slug: '...' }`.
